# 变更提案: 修复订阅名显示错误（Issue #53/#42）

## 需求背景

当前在 Windows 环境中可复现：应用首次启动后，用户直接进入「节点」面板时，订阅分组名称会显示为一串 `sourceConfigId`（UUID），而不是订阅的 `name`。切换到其他面板再切回「节点」后显示恢复正常。

该问题会造成用户误以为订阅名被“替换为乱码”，影响可信度与可用性。

## 变更内容

1. 进入「节点」面板时，确保订阅配置（`/configs`）已加载或正在加载，从而可用 `sourceConfigId -> name` 映射渲染分组标题。
2. 在订阅配置尚未加载完成时，避免直接展示 UUID，改为更友好的占位文案（加载完成后自动刷新为订阅名）。

## 影响范围

- **模块:** 前端主题（共享 UI 逻辑）
- **文件:** `frontend/theme/_shared/js/app.js`
- **API:** 无新增/变更（仅前端调用既有 `/configs`）
- **数据:** 无

## 核心场景

### 需求: 首次进入节点面板订阅名显示正常（Issue #53）
**模块:** 前端主题

#### 场景: 启动后直接进入「节点」面板
应用启动后，用户不进入「订阅」面板，直接点击「节点」面板。
- 预期结果：订阅分组标题显示订阅 `name`，而非 UUID
- 预期结果：无需切换面板即可自动恢复正确显示

#### 场景: 订阅配置加载较慢
进入「节点」面板时订阅配置尚未完成加载。
- 预期结果：分组标题不直接显示 UUID，而显示“加载中/订阅”等友好占位
- 预期结果：订阅配置加载完成后，节点分组标题自动更新为订阅 `name`

## 风险评估

- **风险:** 增加一次对 `/configs` 的调用时机，可能带来轻微的请求开销或在后端不可用时产生额外错误提示。
- **缓解:** 仅在「节点」面板需要时触发加载，并做“已加载”状态缓存；失败时保留原逻辑回退（可显示 ID 以便排障）。
